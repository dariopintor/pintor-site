Git: instalação, comandos e conceitos

Aprenda a versionar seu código com segurança e nunca mais perca seu trabalho

O Git foi desenvolvido por Linus Torvalds para o Linux, mas se tornou tão bom que hoje é o padrão da indústria. Praticamente todo programador no mundo usa. Quando você for procurar emprego, Git estará em quase todo anúncio de desenvolvedor.

Já fizemos uma pequena introdução sobre o que Git é. Agora vamos aprender o básico dele. Para este artigo, vamos focar no Git via terminal — é aí que a mágica acontece.

1. Instalando Git no seu computador

Antes de começar, você precisa instalar Git no seu computador. É um programa como qualquer outro. Leva 5 minutos.

Se você usa Linux, abra o terminal e digite:

sudo apt-get install git

O Linux vai baixar e instalar tudo sozinho. Depois, para ter certeza que funcionou, digite:

git --version

Se aparecer algo tipo "git version 2.40.0", parabéns! Git está instalado.

Se você usa Windows, acesse https://git-scm.com/download/win, baixe o instalador e clique em "Próximo, Próximo, Concluir" — é como instalar qualquer programa normal. Se tiver dúvida em alguma tela, é seguro clicar em "Próximo". Depois que terminar, abra o terminal do Windows (PowerShell ou Command Prompt) e digite git --version. Se aparecer a versão, está tudo certo.

2. Iniciando seu primeiro projeto com Git

Agora que Git está instalado, vamos começar de verdade. Você vai criar uma pasta para seu projeto e "ativar" Git nela.

Comece criando uma pasta em qualquer lugar do seu computador. Pode ser no Desktop, Documents, onde quiser. Dê um nome significativo, tipo "meu-primeiro-projeto" ou "calculadora-js".

Em seguida, você precisa abrir um terminal dentro dessa pasta. Se estiver usando VS Code (que é super recomendado), é bem simples: abra a pasta que você criou diretamente no editor e acesse Terminal > New Terminal no menu superior. Pronto, o terminal fica ali mesmo dentro do editor.

Se preferir usar o terminal externo do seu sistema, navegue até a pasta digitando:

cd caminho/da/sua/pasta

Substitua "caminho/da/sua/pasta" pelo caminho real. Por exemplo: cd Desktop/meu-primeiro-projeto.

Agora, aqui vem o comando mais importante deste passo. Você vai dizer ao Git: "A partir de agora, você é responsável por guardar o histórico desta pasta". Você faz digitando:

git init

Isso cria um diretório oculto chamado .git que Git usa para guardar todo o histórico. Você nunca mexe nele. Git cuida sozinho. Pronto! Seu projeto agora está rastreado por Git. Toda mudança será registrada.

3. Vendo o status do seu projeto

Você criou um arquivo novo chamado index.html na pasta. Como Git sabe disso? Usando o comando git status.

Pense em git status como uma pergunta: "Ei Git, o que mudou desde a última vez que fiz um checkpoint?". Digite:

git status

Git vai responder com algo assim:

On branch master
No commits yet
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        index.html

Git está dizendo: "Tem um arquivo novo aqui (index.html) que você não está acompanhando ainda". É como um aviso de que tem coisas novas.

4. Adicionando arquivos para rastreamento

Agora você disse para Git: "Ei, tem arquivo novo". Mas você ainda não pediu para Git registrar esse arquivo no histórico. Esse é o trabalho de git add.

Pense em git add assim: você está montando uma caixa de presente. Você coloca um item por um dentro da caixa. git add é você colocando um arquivo (ou vários) nessa caixa que depois será lacrada e guardada no histórico.

Para adicionar um arquivo específico, digite:

git add index.html

Isso diz: "Git, rastreie esse arquivo a partir de agora". Se você quer adicionar todos os arquivos da pasta de uma vez, use:

git add *

O asterisco é um coringa que significa "todos os arquivos". Prático quando você mudou vários arquivos de uma vez.

Se você digitar git status novamente, Git vai dizer:

On branch master
Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   index.html

Isso significa: "Ok, você colocou index.html na caixa. Está pronto para ser lacrado no próximo passo".

5. Fazendo seu primeiro commit

Um commit é, basicamente, um "ponto de salvamento" do seu projeto. Na prática, o commit serve para criar um histórico seguro, organizado e rastreável da evolução do seu trabalho. Quando você cria um commit, está criando uma versão específica do seu código. Caso algo dê errado no futuro, você pode facilmente voltar para essa versão.

Toda vez que você faz um commit, você deve deixar uma mensagem explicando o que você fez. Essa mensagem é super importante. Daqui a meses, você vai olhar o histórico e pensar: "O que eu fiz naquele dia?". A mensagem responde isso.

Para commitar (salvar) as mudanças atuais do seu projeto, utilize:

git commit -m "Adicionar a função de login no sistema"

Aquela parte entre aspas é a mensagem do commit. Ela explica exatamente o que você fez. Depois que você executa o comando, o Git vai responder na tela indicando que tudo funcionou, com algo parecido com isso:

[master (root-commit) a1b2c3d] Adicionar a função de login no sistema
 1 file changed, 5 insertions(+)
 create mode 100644 index.html

Isso significa: "Ok, guardei! Essa é a versão a1b2c3d do seu código, registrada com a mensagem que você deixou". Pronto! Você acabou de fazer seu primeiro versionamento. Seu projeto agora tem uma história. Se você mudar os arquivos amanhã e bater o arrependimento, você tem a segurança de poder voltar a esse ponto.

6. Experimentando com segurança: branches

Até agora você trabalhou em uma única "linha" de desenvolvimento. Tudo na branch master (ou main). Mas e se você quer fazer um experimento arriscado? Quer adicionar uma funcionalidade nova mas não sabe se vai dar certo? Não quer mexer no código que já está funcionando?

É aí que entra branches. Pense em branch como um "universo paralelo" do seu código. Você cria um branch, faz mudanças lá, testa, e se funcionar, você "mescla" de volta com o master. Se não funcionar, você simplesmente descarta.

Digamos que você está desenvolvendo uma calculadora e quer adicionar a função de divisão. Você cria um branch chamado "feature/divisao":

git checkout -b feature/divisao

Esse comando cria um branch novo e já te coloca dentro dele. Agora tudo que você fazer (git add, git commit) acontece nesse branch paralelo, não no master. Você trabalha normalmente: edita arquivos, faz git add, git commit. Tudo funciona igual.

Se você quer ver em qual branch você está agora, digite:

git branch

Git mostra a lista de branches que existem e marca qual você está agora com um asterisco *.

Quando terminou de testar no branch "feature/divisao" e quer voltar ao master, você faz:

git checkout master

Isso "troca de universo". Você volta ao código do master, que não foi tocado pelas mudanças que você fez no outro branch.

Se tudo funcionou e você quer trazer as mudanças do branch "feature/divisao" para o master, você faz um merge (fusão):

git merge feature/divisao

Isso diz: "Git, pegue tudo que eu fiz no branch feature/divisao e traga para o master". Se não tiver conflito, pronto! Está mesclado. A beleza disso é que você e um colega podem trabalhar em branches diferentes. Você na "feature/divisao", ele na "feature/subtracao". Ninguém mexe no código um do outro. Depois vocês mesclam tudo no master.

Branches são super poderosas. Com elas, você não tem medo de experimentar. Quer tentar uma abordagem diferente? Cria um branch. Não funcionou? Volta ao master. Funcionou? Mescla. Simples assim.

7. Enviando seu código para o mundo

Você já aprendeu tudo de Git. Seu código está versionado localmente. Cada mudança está registrada. Você pode voltar para qualquer ponto no tempo.

Mas e se seu computador quebrar? E se você quer trabalhar em vários computadores? E se você quer mostrar seu código para outras pessoas?

É aí que entra git push. Esse comando envia seu código (todos os commits) para um servidor remoto. A ferramenta mais popular para isso é GitHub. No momento em que você vai usar git push pela primeira vez, você vai precisar de uma conta no GitHub. Se não tem, crie uma em https://github.com/signup. É gratuito. Depois que criar a conta, você conectará seu computador local com sua conta lá. A partir daí, você consegue fazer:

Sincronizando seu Código com o GitHub

Após realizar as alterações no seu projeto local, você precisa sincronizá-las com o servidor remoto. Para isso, utilizamos dois comandos principais: o Push e o Pull.

Enviando arquivos: git push

O comando git push serve para enviar suas alterações locais para o repositório remoto, garantindo que seu código esteja seguro na nuvem e acessível de qualquer lugar.

  • Como utilizar: No terminal, informe o destino utilizando os parâmetros origin (o servidor remoto) e master (a sua ramificação principal).
git push origin master
  • Autenticação: O terminal solicitará seu usuário e senha (ou token de acesso) do GitHub para validar a operação.
  • Resultado: Seu código é armazenado com segurança, evitando perdas caso algo aconteça com seu computador.

Recebendo atualizações: git pull

Enquanto o push envia, o git pull traz as mudanças que estão no servidor para o seu computador.

Exemplo prático:

Imagine que você e um colega estão colaborando no mesmo projeto. Se ele finalizar uma funcionalidade e fizer um git push, a versão do servidor estará mais atualizada que a sua. Para continuar o trabalho de onde ele parou e garantir que você tenha a versão mais recente, você deve executar o git pull.

Resumo Técnico:

  • Push: Upload do código local → Servidor Remoto.
  • Pull: Download do código do Servidor → Máquina Local.
  • Dica: Sempre é boa prática fazer um pull antes de começar a trabalhar para evitar conflitos de versão.

8. Seu fluxo de trabalho, na prática

Depois de tudo que você aprendeu, aqui está como o seu dia a dia vai funcionar quando estiver desenvolvendo.

Você chega para trabalhar no seu projeto. A primeira coisa é fazer git pull para trazer qualquer mudança que possa ter vindo do servidor (se estiver trabalhando em time) ou simplesmente começar direto no seu editor. Você edita seus arquivos normalmente, criando novas funcionalidades ou corrigindo bugs.

De tempos em tempos — quando você termina uma funcionalidade, ou resolve um problema, ou qualquer mudança significativa — você faz git status para ver o que mudou. Depois você usa git add * para preparar todos esses arquivos para serem salvos. Logo após, faz git commit -m "sua mensagem" deixando claro o que você fez naquele ponto.

Repete esse ciclo quantas vezes quiser ao longo do dia. Nada de perder trabalho. Nada de pânico. Cada mudança importante está guardada em um ponto seguro.

Se você quer experimentar algo arriscado, cria um branch. Trabalha à vontade nesse universo paralelo. Se deu certo, volta ao master e faz um merge para trazer as mudanças de volta. Se não deu certo, você simplesmente volta ao master e aquele experimento fica guardado no histórico, mas não interfere no seu código principal.

Quando você termina uma sessão de trabalho e quer que seu código esteja seguro na nuvem (ou quer compartilhar com um colega), faz git push e seu trabalho sobe para o GitHub.

Isso é Git. E com isso você nunca mais perde código. Nunca mais entra em pânico. E está pronto para trabalhar em time sem conflitos.
