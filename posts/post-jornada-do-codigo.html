Descomplicando o vocabulário do desenvolvimento de software

Um guia prático com os termos essenciais para você entender a área tech.

Uma coisa que sempre me preocupa é como meus alunos estão aprendendo. Estou sempre buscando formas de melhorar minhas analogias e práticas em sala de aula, a fim de manter o interesse deles na disciplina. Algo que sempre tenho em mente é que, para a aprendizagem se tornar mais eficaz, o conhecimento precisa ter significado. Isso quer dizer que, à medida que aprendemos algo novo e o relacionamos com coisas que já sabemos, vamos dando sentido ao que estamos aprendendo, o que torna o processo mais fácil e duradouro. Quem nos ensina isso é o nosso grande educador Paulo Freire, que desenvolveu um método de alfabetização para adultos no qual o ensino se fundamentava em 'palavras geradoras' extraídas da realidade do estudante, significando assim os novos conceitos.

Mas, trazendo essa forma de ensinar para o nosso mundo da tecnologia, como posso encontrar essas 'palavras geradoras' para os meus alunos? Alguns já chegam com uma boa bagagem de termos do universo tech, mas, como é praxe na docência, não partimos do princípio de que todos estão no mesmo nível. Pensando nisso, resolvi escrever este texto geral abordando o passo a passo para desenvolver um software (formalmente conhecido como processo de software), para já ir introduzindo palavras e conceitos que ajudarão os alunos iniciantes.

O artigo estará organizado com um texto geral explicando o passo a passo, tentando oferecer uma 'visão de cima da floresta'. Em cada tópico, haverá links para artigos mais aprofundados sobre o tema, oferecendo assim uma 'visão de dentro das árvores'.

Você não precisa decorar nada disso agora. O objetivo é você começar a reconhecer essas palavras-chave quando ouvir um professor ou colega falando sobre elas e entender que elas são peças diferentes de um quebra-cabeça maior. Assim, quando mergulharmos nos detalhes de cada uma dessas etapas nos próximos artigos, você já terá uma base para compreender o que está acontecendo. Lembre-se sempre: todo software, grande ou pequeno, segue esse caminho. E você, ao compreender isso, já está alguns passos à frente na sua jornada de aprendizado.

Organizando o Projeto: O Processo de Software

Imagine que você quer construir uma casa. Você não chega no lote, pega uma enxada e começa a cavar sem direção, certo? Antes disso, você precisa de um plano, um mapa que mostre por onde começar. Pois bem, o processo de software é exatamente essa bússola. É o mapa que nos guia desde o momento em que temos uma ideia até o instante em que o software está pronto e funcionando nas mãos dos usuários. Um processo de software nada mais é do que a organização das etapas de desenvolvimento de um software

Quando estamos iniciando na programação, pode parecer que um processo é "chato" ou "desnecessário". Afinal, para programas pequenos e pessoais, você pode simplesmente começar a codar e ir ajustando. Mas aqui está a verdade: à medida que os projetos crescem, a falta de um processo vira caos. Geralmente vemos sobre processo de software nas disciplinas de análise e projeto nos cursos de Tecnologia da Informação, que acontece do segundo modulo em diante. Existem várias formas de organizar o processo de software, mas de forma bem geral, todo software passa por estas etapas: análise, quando você entende o que o cliente quer; design, quando você planeja como vai construir tudo isso; desenvolvimento, que é quando a gente codifica de verdade; testes, quando você verifica se tudo funciona e encontra os erros; deployment, que é apenas um nome chique para "colocar no ar" para as pessoas usarem; e manutenção, quando você cuida, atualiza e melhora o software mesmo depois dele estar pronto.

Arquitetura de Software: O Projeto da Casa Antes de Construir

Lembra quando falamos que o processo de software é como ter um mapa para construir uma casa? Pois bem, dentro desse mapa existe algo muito importante chamado arquitetura de software. A arquitetura é o projeto detalhado de como as peças da sua aplicação vão estar organizadas e como vão se comunicar entre si. Não é sobre qual tecnologia usar (HTML, JavaScript, Python...), mas sim sobre qual estrutura, qual formato você vai dar ao seu software. Pense assim: a arquitetura é o projeto do arquiteto (como os cômodos estão organizados, como as pessoas se movem), enquanto as tecnologias são os materiais que você usa (tijolos, cimento, madeira). Um bom projeto com materiais ruins é melhor do que um projeto ruim com materiais caros!

Existem vários tipos de arquitetura, cada um com um propósito diferente. A arquitetura cliente-servidor é a mais comum e é exatamente o que você já conhece: você (cliente) faz requisições para um servidor central que processa tudo e responde (é como um restaurante onde você pede ao garçom e ele vai à cozinha). A arquitetura P2P (peer-to-peer, ou "ponto a ponto") é quando todos os computadores são iguais e se comunicam diretamente uns com os outros, sem um chefe (é como amigos compartilhando arquivos direto pelo WhatsApp, sem um intermediário) – funciona bem para apps de compartilhamento, mas é mais complexa. A arquitetura monolítica é quando tudo está junto em um único bloco (como uma casa com apenas um grande salão em vez de cômodos separados) – é simples para pequenos projetos, mas fica bagunçada conforme cresce. A arquitetura de microsserviços divide a aplicação em pequenos serviços independentes que conversam entre si (como cômodos separados que funcionam independentemente) – é mais complexa, mas muito mais flexível para grandes projetos. A arquitetura em camadas (ou MVC) organiza seu código em "andares": a camada de apresentação (frontend), a camada de lógica (backend) e a camada de dados (banco de dados) – é um meio termo que funciona bem para a maioria dos projetos iniciantes.

A escolha da arquitetura certa depende do tamanho do seu projeto, de quantas pessoas estão trabalhando, e de quanto tempo você quer que ele dure sem precisar ser reescrito. Um app pequeno pode usar cliente-servidor simples ou monolítico, mas um gigante como Netflix usa microsserviços para conseguir escalar, enquanto apps como BitTorrent usam P2P para distribuir dados. Essas decisões geralmente acontecem na fase de design do processo de software – é aí que você, junto com o time, decide qual será o "esqueleto" da sua aplicação. Não se preocupe em escolher a arquitetura perfeita agora – você aprenderá mais sobre isso nas disciplinas de Desenvolvimento Web e em cursos mais avançados. Por enquanto, o importante é saber que escolher a arquitetura certa é tão importante quanto escolher as tecnologias!

A Anatomia da Aplicação: Frontend, Backend e Banco de Dados

Toda aplicação que você usa tem três componentes principais que trabalham juntos: frontend, Backend e Banco de Dados. Pense em um restaurante: o frontend é o garçom que você vê (tudo que aparece na sua tela – botões, imagens, cores) e é construído com tecnologias como HTML, CSS e JavaScript, ou frameworks como React e Vue; o backend é a cozinha nos bastidores (a lógica que processa seus dados) feito com linguagens como Python, Java, Node.js ou C#; e o banco de dados é a dispensa onde todos os ingredientes ficam guardados e organizados, usando tecnologias como MySQL, PostgreSQL ou MongoDB. Quando você clica em um botão no Instagram, por exemplo, o frontend recebe sua ação, envia para o backend processar, que consulta o banco de dados e devolve a resposta para o frontend mostrar na tela.

Você só vê e interage com o frontend, mas nos bastidores o backend está o tempo todo verificando se você tem permissão, validando informações e consultando o banco de dados. É por isso que sua senha não fica armazenada no seu celular – ela fica guardada de forma segura no banco de dados, protegida e acessível de qualquer dispositivo. Sem essas três peças trabalhando juntas, nada funcionaria.

A grade do curso do TDS (Técnico de Desenvolvimento de Software) do IFB foi pensada justamente para você aprender essas peças de forma progressiva! Na disciplina de Lógica de Programação I, você aprenderá JavaScript e na de Webdesign, estudará HTML e CSS – tudo que você precisa para começar com frontend. Os frameworks de front (como React e Vue) virão depois, na disciplina de Desenvolvimento Web. Já a parte de backend é abordada em Lógica de Programação II. Então não se preocupe em aprender tudo agora – você começará com frontend (mais visual e motivador), depois backend (onde a lógica fica mais complexa), e por fim banco de dados (mais técnico).

Qualidade e Teste de Software

A qualidade de sofware é levada em consideração durante todo ciclo de desenvolvimento de sofware (processo de software), porém, o Teste de Software possui um fase específica dentro do processo. O objetivo dessa etapa não é apenas caçar "bugs" ou apontar erros, mas sim garantir que a aplicação funcione exatamente como o esperado em diferentes cenários, proporcionando uma experiência segura, rápida e sem frustrações para o usuário final.

Para criar essa segurança, os desenvolvedores escrevem "códigos que testam outros códigos", criando uma verdadeira rede de proteção automatizada. Existem os testes de unidade, que verificam se uma pequena engrenagem do sistema funciona sozinha -- quando você está fazendo código pequenos para uma disciplina de programação e entra com dados e observando se a saída está correta você já está fazendo testes unitários, porém de forma bem artezanal; os testes de integração, que garantem que o frontend e o backend, por exemplo, consigam conversar sem ruídos; e até os testes de sistema, que simulam o comportamento de um usuário real clicando, preenchendo formulários e navegando pela aplicação.

O Controle do Caos: Versionamento de Código

Agora que você já sabe como as três peças de uma aplicação se encaixam, surge uma pergunta natural: e quando várias pessoas estão mexendo no mesmo código ao mesmo tempo? Ou quando você faz uma mudança e percebe, horas depois, que ela quebrou tudo? É para resolver exatamente esse problema que existe o versionamento de código. Versionamento nada mais é do que um sistema que registra cada alteração feita no código ao longo do tempo, como se fosse um histórico detalhado de tudo que aconteceu no projeto. A ferramenta mais usada para isso no mundo todo é o Git, e junto com ela, plataformas como o GitHub funcionam como uma nuvem onde esse histórico fica armazenado e compartilhado com o time.

Pense no versionamento como o "Ctrl+Z do projeto inteiro". Se você e um colega trabalharam juntos no frontend e no backend ao mesmo tempo e algo deu errado, o Git permite que você volte a qualquer ponto anterior do projeto sem perder o trabalho de ninguém. Além disso, ele organiza o trabalho em equipe por meio de branches (ramificações), onde cada pessoa pode desenvolver uma funcionalidade de forma independente, sem atrapalhar o código dos outros, e depois unir tudo de forma controlada. No deploy e no fluxo de CI/CD, o versionamento é o ponto de partida: é a partir do código versionado e enviado para o GitHub que todas as etapas de teste e publicação automática são disparadas.

A Entrega: O que é o Famoso Deploy?

Até aqui já sabemos que para desenvolver um software precisamos de um processo e também que durante todas as etapas desse processo precisamos nos preoucupar com a qualidade desse software e por isso existe o teste de software. Aprendemos ainda que uma aplicação é organizada em backend e frontend e banco de dados e que por uma questão de organização do nosso trabalho de codificação existe o versionamento de código. Agora vamos saber como entregar o que nós fizemos ao cliente final.

O Deploy é simplesmente o ato de colocar sua aplicação online, em um servidor, para que as pessoas consigam acessá-la. Ele não é apenas "copiar e colar" seus arquivos de código em um servidor qualquer. Para que esse processo seja seguro e rápido, utilizamos a cultura DevOps (Integração entre Desenvolvimento e Operações). O objetivo é automatizar tudo através de um fluxo chamado CI/CD. O CI (Integração Contínua) garante que cada pequena mudança no código seja testada automaticamente para evitar erros. Já o CD (Entrega Contínua) é o que faz o trabalho pesado de levar essas mudanças para o servidor de forma automática, sem que você precise fazer isso manualmente toda vez.

Existem vários passos importantes nesse fluxo: você precisa preparar seu código, garantir que todas as dependências estão corretas, configurar as variáveis de ambiente (informações secretas como senhas) e validar se tudo está funcionando. Tecnologias como Vercel e Netlify (para frontend), Heroku, AWS e Azure (para backend), Docker (para empacotar a aplicação) e GitHub Actions (para gerenciar o CI/CD) são as ferramentas que ajudam a fazer essa "mágica" acontecer no mundo real.

Como você viu existem várias etapas no deploy que inclusive conversam com o processo de software. Porém, agora, o importante é você entender que desenvolver, testar e fazer deploy são fases diferentes e igualmente importantes no processo de criar software. Quando você terminar seu primeiro projeto e conseguir colocar ele online de forma automatizada para seus amigos acessarem, aí sim você sentirá a verdadeira magia da programação!

Bibliotecas, frameworks e APIs

Você já parou para pensar no quanto seria complicado criar um site inteiro se tivesse que escrever cada linha de código do zero? Pois bem, existe uma solução elegante para esses problemas: bibliotecas, frameworks e APIs. Essas ferramentas são responsáveis por fazer a vida de qualquer desenvolvedor ser muito mais prática e você vai usá-las em praticamente todo projeto real que trabalhar.

A sua caixa de ferramentas

Uma biblioteca é uma coleção de recursos e funções pré-escritas que você pode "chamar" sempre que precisar resolver um problema específico, como formatar uma data ou realizar um cálculo complexo. Em JS temos a biblioteca Day.js (para manipular datas). Se você precisa formatar uma data complexa, você chama a biblioteca, ela faz o cálculo e te devolve o resultado. Aqui tecnicamente, o controle é seu: o seu código importa a biblioteca e chama as funções dela no momento em que você decidir.

A casa pré-fabricada

Já um framework é um conjunto estruturado de ferramentas, bibliotecas e padrões que fornece uma base pronta — um verdadeiro "esqueleto" — para o desenvolvimento de aplicações. Mais do que apenas facilitar o trabalho, ele estabelece a arquitetura geral, convenções e fluxos que você deve seguir. É aqui que ocorre o que chamamos tecnicamente de Inversão de Controle: você não apenas chama (como na biblioteca) o framework para resolver um problema isolado, mas é aqui que ele defineo framework que dita o ambiente de execução e "chama" o seu código nos lugares e momentos corretos. Exemplos de frameworks no universon JS são o Next.js e Vue.js para o front-end, Node.js com Express para back-end, e Django ou Flask em Python.

O Porteiro e as Conexões

Uma API (Application Programming Interface) é um conjunto de regras e especificações que permite que diferentes softwares se comuniquem entre si. A API funciona como o porteiro de seu prédio: imagine que alguém queira acessar o seu apartamento dizendo que é um eletricista ou encanador (um sistema ou aplicativo externo tentando ler ou modificar as informações no seu banco de dados). Em vez de você deixar a porta da rua aberta para qualquer um subir direto e mexer nas suas coisas, o prédio tem o porteiro. Ele segue um rígido protocolo de segurança: barra o prestador de serviço na portaria e exige a identificação correta e a ordem de serviço. Ele só deixa entrar quem tem autorização, e mesmo assim, ele só pode acessar as áreas específicas do prédio que estão autorizadas. Assim funciona uma API: ela é a camada de segurança e controle que permite que outros sistemas acessem apenas o que você quer que eles acessem, seguindo regras pré-definidas. No dia a dia, você usará APIs de duas formas:

  • APIs de terceiros: você pode usar uma API já pronta desenvolvida por teceiros. Por exemplo, usar a Fetch API do JavaScript para buscar dados externos, como a previsão do tempo ou preços de ações, para seu sistema
  • Usando sua própria API: Em projetos reais, você cria sua própria API (usando ferramentas como Node.js com Express) para permitir que seu front-end acesse com segurança os dados salvos no seu banco de dados. Em vez do site acessar o banco diretamente, ele faz uma requisição para a sua API, que valida o acesso e entrega as informações necessárias. Nesse contexto sua API nada mais é do que o seu backend.

    • A diferença entre os três

    A principal diferença entre os três conceitos reside no controle do fluxo de execução e no papel estrutural que cada um desempenha no projeto: a biblioteca é uma ferramenta específica que você chama ativamente apenas quando precisa resolver um problema pontual, mantendo o controle total do código nas suas mãos; o framework inverte essa lógica ao fornecer a base arquitetônica da aplicação, ditando as regras de onde e como o seu código deve ser inserido para que ele assuma o controle e o execute; e, por fim, a API funciona como o canal de comunicação — o "porteiro" rigoroso —, estabelecendo contratos e regras de segurança para que essa aplicação estruturada pelo framework possa trocar dados de forma segura e padronizada com sistemas externos ou com o seu próprio banco de dados.

    A importância de utilizar frameworks e APIs está na eficiência, segurança e escalabilidade do desenvolvimento. As bibliotecas atuam como caixas de ferramentas especializadas, poupando o esforço de codificar funções complexas e rotineiras do zero, mantendo o controle nas mãos do desenvolvedor. Os frameworks eliminam a necessidade de "reinventar a roda", acelerando o desenvolvimento ao fornecer soluções já testadas e otimizadas. APIs permitem reutilizar funcionalidades existentes, reduzindo tempo de desenvolvimento e possibilidades de erro, além de possibilitar a integração com serviços especializados sem necessidade de desenvolvê-los do zero.
